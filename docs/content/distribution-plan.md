# DreamCoder 首发与增长计划

## 定位与受众

DreamCoder 的首要受众不是“想一键做商业游戏的所有人”，而是中国开发者社区中的初中级全栈、Python 与 AI 应用开发者、学生和技术作者。

可验证定位：

> 一个开源、自托管的 AI Web 小游戏生成工作台和完整 AI 应用参考实现，覆盖描述、生成文件、安全预览与基于现有文件继续修改。

不使用“企业级”“通用 AI IDE”“零代码商业游戏平台”等当前无法证明的表达。

## 首发前的最小证据包

- [x] 中英文 README 与十分钟 Quickstart
- [x] CI、测试、前端构建和依赖审计
- [x] 三个可离线试玩的确定性示例
- [x] 当前限制、安全说明和 Roadmap
- [ ] 一次真实 provider 的“创建 → 预览 → 继续修改”录屏
- [ ] GitHub `v0.1.0` Release

前四项让读者能评估工程质量；真实录屏证明核心产品链路。没有真实录屏时，不制作伪造生成过程。

## 首发顺序

### T0：GitHub

1. 确认 README 首图和真实演示；
2. 合并并确保 CI 绿色；
3. 发布 `v0.1.0`；
4. 创建两个 Issue：跨平台 Quickstart 反馈、示例游戏征集；
5. 视维护精力决定是否开启 Discussions。

### T+1 天：CSDN 首发

发布 `csdn-launch-draft.md`，重点不是项目公告，而是三个可搜索的工程问题：

- LangGraph 如何基于 existing files 继续生成；
- 生成任务失败后如何正确收尾；
- AI 生成 HTML 如何做路径校验、CSP 和 iframe sandbox。

主 CTA 只有一个：访问 GitHub 并跑 Quickstart。不要同时要求关注多个账号、进群和私信。

### T+3 至 T+7 天：二次分发

- 掘金：重写开头和标题，保留技术拆解，使用 GitHub 作为原始项目链接；
- 知乎：回答“LangGraph 有哪些完整项目”或“AI 生成网页如何安全预览”，避免纯链接帖；
- B 站：3–5 分钟演示视频，前 15 秒展示输入、成品和继续修改；
- GitHub：把视频/GIF 加回 README，发布首发文章链接到 Discussions（若已开启）。

不要同一天原文复制到所有平台。错开 2–4 天并调整标题、开头和 CTA，便于判断每个平台的真实流量。

## SEO 执行

- 标题包含一个明确技术问题与 2–4 个核心技术词，不堆满所有框架名。
- 前 150 字直接回答“项目是什么、能做什么、读者会学到什么”。
- 可复制命令使用真实文本代码块，不把关键步骤做成图片。
- 错误排查文章把完整错误文本放进标题、H1 与首屏。
- 每篇包含“适用版本/最后更新：2026-07-23”，模型 ID 相关内容按季度复查。
- 从 GitHub Issues、CSDN 评论和搜索联想收集真实问题，再决定后续关键词。

## GEO 执行

GEO 的目标是让搜索摘要和回答系统可以准确理解、引用项目，而不是机械重复关键词。

- 使用稳定的一句话定义，并在 README、Release 和文章中保持一致；
- 为关键决策提供“需求 → 选择 → 何时替换”的表格；
- 明确区分已实现、可选、计划中和不支持；
- 给出可核验的文件路径、命令、版本日期和限制；
- 使用 FAQ 风格回答“需要 Docker 吗”“支持哪些模型”“生成代码安全吗”；
- 链接官方 provider 文档与项目源码，避免没有来源的性能数字；
- 发布后用真实 Issue、测试结果和版本说明更新内容，建立时间连续性。

## 视频脚本（3–5 分钟）

1. 0:00–0:15：直接展示一句需求、可玩的结果、第二句修改；
2. 0:15–0:45：一句话定位和适合谁；
3. 0:45–1:30：本地启动只需 Python、Node、SQLite 与一个 Key；
4. 1:30–2:30：工作流与 existing files；
5. 2:30–3:30：路径校验、CSP、sandbox 和真实限制；
6. 3:30–4:00：三个离线示例与 GitHub CTA。

画面必须来自真实运行。模型等待时间可以倍速，但应标注；失败过程可以保留一小段，增加可信度。

## 衡量指标

首月不以 Star 总数作为唯一目标，记录完整漏斗：

| 层级 | 指标 | 数据来源 |
|---|---|---|
| 曝光 | 阅读、播放、搜索展示 | CSDN/B 站/Search Console |
| 兴趣 | GitHub 独立访客、README 浏览 | GitHub Traffic |
| 尝试 | Clone、示例访问、Quickstart Issue | GitHub Traffic/Issues |
| 激活 | 首次生成成功、完成一次继续修改 | 自愿反馈模板 |
| 社区 | Star、Issue、PR、示例投稿 | GitHub |

发布当天、7 天和 30 天各记录一次。若文章有阅读但 GitHub 点击少，改首屏与 CTA；若 Clone 多但反馈卡在安装，优先修 Quickstart；若首次生成成功但无人继续修改，重新验证核心价值定位。
